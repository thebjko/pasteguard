# 인수인계: stream-transformer 버퍼 공유 버그

## 문제 요약

`src/providers/anthropic/stream-transformer.ts`에서 `piiBuffer`와 `secretsBuffer`가
`text_delta`와 `input_json_delta` 처리 시 **같은 변수를 공유**한다.

---

## 실제 버그 시나리오

Anthropic 스트리밍 응답 한 건 안에서 text block + tool_use block이 순서대로 올 수 있다.

```
content_block_start (text)
  content_block_delta: text_delta "...010-2345-[[KOREAN_"   ← 플레이스홀더가 잘림
content_block_stop                                          ← 여기서 버퍼를 안 비움
content_block_start (tool_use)
  content_block_delta: input_json_delta "PHONE_1]]..."      ← 이전 버퍼 오염
```

### 현재 동작 (버그)

1. `text_delta` 처리 중 `[[KOREAN_` 가 `secretsBuffer`에 남음
2. `content_block_stop` 이벤트 → 현재 코드는 버퍼를 플러시하지 않고 그냥 통과
3. `input_json_delta` 처리 시 `secretsBuffer`에 남은 `[[KOREAN_` + `PHONE_1]]` 가 합쳐져서
   **tool input의 JSON 조각 안에 언마스킹된 전화번호가 섞여 들어감**

실제로 플레이스홀더가 content block 경계를 넘어 걸칠 확률은 낮지만,
응답이 긴 경우 충분히 발생할 수 있다.

---

## 다중 에이전트 동시 사용 관련

**요청 간 버퍼 공유는 없다.** `piiBuffer`, `secretsBuffer`, `lineBuffer`는
모두 `createAnthropicUnmaskingStream()` 클로저 내 지역 변수이므로
요청마다 독립적으로 생성된다.

즉, 여러 Claude agent가 동시에 서버를 사용해도 서로의 버퍼를 침범하지 않는다.
문제는 **하나의 스트림 응답 내부**에서만 발생한다.

---

## 수정 방법

`content_block_stop` 이벤트를 감지해서 버퍼를 플러시하고 리셋한다.

### 수정 대상 파일

`src/providers/anthropic/stream-transformer.ts`

### 수정 위치

`parsed.type === "content_block_delta"` 를 처리하는 else 분기 바로 위에 추가:

```typescript
// 현재 (버그 있음):
} else {
  // Pass through non-delta events unchanged
  controller.enqueue(encoder.encode(`data: ${data}\n`));
}

// 수정 후:
} else {
  // content_block_stop 시 버퍼 플러시 및 리셋
  if (parsed.type === "content_block_stop") {
    let flushed = "";
    if (piiBuffer) {
      flushed += piiContext
        ? flushMaskingBuffer(piiBuffer, piiContext, config)
        : piiBuffer;
      piiBuffer = "";
    }
    if (secretsBuffer) {
      flushed += secretsContext
        ? flushSecretsMaskingBuffer(secretsBuffer, secretsContext)
        : secretsBuffer;
      secretsBuffer = "";
    }
    if (flushed) {
      // 남은 버퍼를 text_delta로 먼저 내보낸 뒤 block_stop 전달
      const flushEvent = {
        type: "content_block_delta",
        index: (parsed as { index?: number }).index ?? 0,
        delta: { type: "text_delta", text: flushed },
      };
      controller.enqueue(encoder.encode(`data: ${JSON.stringify(flushEvent)}\n`));
    }
  }
  controller.enqueue(encoder.encode(`data: ${data}\n`));
}
```

### 완료 시 버퍼 플러시 (done 처리) 도 함께 수정

현재 done 처리에서 `piiBuffer`와 `secretsBuffer`를 합산해서 하나의 text_delta로 보내는데,
content_block_stop에서 미리 플러시하면 done 시점에 버퍼가 비어있게 되어 해당 코드는 사실상 무동작이 된다.

## 추가 버그: event 라인 선발행으로 인한 SSE 파싱 에러

### 증상

```
Could not parse message into JSON:
From chunk: [ "event: content_block_delta" ]
```

claude CLI가 SSE 스트림을 파싱할 때 위 경고가 발생한다.
응답 자체는 오지만 노이즈가 발생하고, 특정 청크가 유실될 수 있다.

### 원인

`stream-transformer.ts`에서 `event:` 라인과 `data:` 라인을 **개별적으로** enqueue한다.

```typescript
// 83~85줄: event 라인 즉시 발행
if (line.startsWith("event: ")) {
  controller.enqueue(encoder.encode(`${line}\n`));
  continue;
}

// 126줄: processedText가 비어있으면 data 라인 발행 안 함
if (processedText) {
  controller.enqueue(encoder.encode(`data: ${JSON.stringify(modifiedEvent)}\n`));
}
```

텍스트 청크가 버퍼에 **전부** 흡수되면 (예: 플레이스홀더의 시작 부분만 포함된 청크):

1. `event: content_block_delta\n` — 발행됨
2. `data: {...}\n` — **발행 안 됨** (processedText가 빈 문자열)
3. `\n` (빈 줄) — 발행됨

결과적으로 SSE 이벤트 그룹이 `event: content_block_delta\n\n`이 되고,
claude CLI는 이 이벤트를 `["event: content_block_delta"]` 형태로 수집한 뒤
JSON 파싱을 시도하다 실패한다.

### 수정 방법

`event:` 라인을 즉시 발행하지 말고 **보류**했다가, `data:` 라인을 발행할 때 함께 내보낸다.
`data:` 라인을 발행하지 않기로 결정했으면 보류한 `event:` 라인도 버린다.

```typescript
// event 라인 변수에 보류
let pendingEventLine = "";

if (line.startsWith("event: ")) {
  pendingEventLine = `${line}\n`;
  continue;
}

if (line.startsWith("data: ")) {
  // ...processedText 계산...

  if (processedText) {
    if (pendingEventLine) controller.enqueue(encoder.encode(pendingEventLine));
    controller.enqueue(encoder.encode(`data: ${JSON.stringify(modifiedEvent)}\n`));
    pendingEventLine = "";
  } else {
    // data를 안 보낼 거면 event도 버린다
    pendingEventLine = "";
  }
  // data를 그대로 통과시키는 경우도 동일하게 처리
}
```

> `pendingEventLine`은 각 이벤트 그룹마다 리셋해야 한다.
> `content_block_stop`처럼 항상 data를 내보내는 이벤트는 기존 방식대로 해도 무방하지만,
> 일관성을 위해 전부 pendingEventLine 패턴으로 통일하는 것을 권장한다.

## 함께 처리해야 할 작업: document 블록 마스킹

Anthropic의 `document` content block은 현재 `UnknownBlockSchema`로 통과만 하고 마스킹이 안 된다.

### 구현 범위

1. **`src/providers/anthropic/types.ts`** — `DocumentBlockSchema` 추가
2. **`src/masking/extractors/anthropic.ts`**
   - `extractTexts`: `source.type === "text"` → `source.data` 추출 (`nestedPartIndex: 0`)
   - `extractTexts`: `title` 추출 (`nestedPartIndex: 1`), `context` 추출 (`nestedPartIndex: 2`)
   - `applyMasked`: document 블록에 마스킹된 텍스트 적용
3. **테스트 추가**

`base64` / `url` source는 바이너리 또는 원격 참조이므로 마스킹 불가, 스킵.
